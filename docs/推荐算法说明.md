# 装备推荐算法说明

## 一、整体流程

推荐采用**三层混合策略**：约束过滤 → 协同过滤 + 内容匹配 → 分类多样性选取。

```
出行计划/用户偏好 → 约束过滤(候选集) → 协同+内容打分 → 分类多样性选取 → 最终推荐列表
```

---

## 二、匹配度（得分）计算

### 1. 约束过滤（第一层）

根据出行计划的**季节、预算、活动类型、人数**筛选候选装备：

| 约束项 | 匹配规则 |
|--------|----------|
| 季节 | 装备的 season 包含计划季节（如计划「春」→ 装备「春,夏,秋」可匹配） |
| 预算 | 装备价格 ≤ 计划预算 × 1.2 |
| 活动类型 | 装备的 scene 包含计划活动（如「徒步」→ 装备 scene 含「徒步,露营」可匹配） |
| 人数 | 装备 maxUsers ≥ 计划人数（或装备无人数限制） |

### 2. 协同过滤得分（30% 权重）

基于用户行为（收藏、评分、加购、购买、**计划选品**）及**相似计划**：
- **相似用户**：① 与当前用户有共同喜欢的装备 ② 或有相似出行计划（季节、活动类型一致）
- **计划选品**：用户加入计划的装备视为强偏好信号
- 统计相似用户喜欢的装备，当前用户未接触过的装备获得协同得分

### 3. 内容匹配得分（40% 权重）

根据计划属性与装备属性计算相似度：

| 匹配项 | 得分规则 |
|--------|----------|
| 标签重叠 | Jaccard 相似度：计划偏好标签 ∩ 装备标签 / 并集 |
| 季节匹配 | 装备 season 包含计划季节 → +1.0 |
| 场景匹配 | 装备 scene 包含计划活动类型 → +1.0 |
| 人数匹配 | 装备 maxUsers ≥ 计划人数 → +0.5 |

计划偏好标签由**活动类型、季节**映射到标签表（如「徒步」→ 标签「徒步」，「春」→ 标签「春」）。

### 4. 基础分（30% 权重）

- 有约束时：1.0
- 无约束时：0.5

### 5. 新装备探索加分（+0.05）

创建时间 7 天内的装备获得额外加分，缓解冷启动。

### 6. 最终得分公式

```
总分 = 0.3 × 协同得分 + 0.4 × 内容得分 + 0.3 × 基础分 + 探索加分
```

---

## 三、分类多样性选取

为避免单一分类（如帐篷）垄断推荐结果：

1. 按**总分**对候选装备排序
2. 按**分类**（帐篷、睡袋、背包等）分组
3. **轮流选取**：第 1 轮从每类取最优 1 个，第 2 轮再取 1 个，依此类推
4. 直到凑满 topN（默认 12 个）

这样可保证推荐列表包含帐篷、睡袋、背包、睡垫等多种装备类型。

---

## 四、缓存

- 推荐结果缓存 30 分钟（Redis）
- 缓存 key 含版本号，算法更新后旧缓存自动失效
- 若未配置 Redis，每次请求实时计算

---

## 五、若仍只看到帐篷

1. **重启后端**：确保新代码已生效
2. **刷新页面**：前端可能缓存了旧数据
3. **检查 Redis**：若使用 Redis，可清空 `recommend:*` 键或等待 30 分钟过期
