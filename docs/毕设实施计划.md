## 阶段二：数据库与领域模型（约 2–3 周）

### 2.1 概念模型与表设计
- [x] **用户与权限**：用户表、角色表、用户-角色关联；字段含账号、密码（加密）、昵称、手机/邮箱、角色、状态、积分、等级等
- [x] **装备**：装备表（名称、分类、品牌、价格、季节、场景、重量、描述、图片等）、装备标签/属性表，支持多维度筛选与推荐（表设计见《阶段二-数据库设计》4.1–4.4）
- [x] **出行计划**：计划表（用户、目的地、时间、人数、预算、季节、活动类型等），用于约束推荐（表设计见《阶段二-数据库设计》5.1–5.2）
- [x] **用户行为**：浏览、收藏、加购、评分、评论等，用于协同过滤与内容推荐（表设计见《阶段二-数据库设计》）
- [ ] **社区**：帖子、评论、点赞、举报等（若开题要求）
- [ ] **系统与运营**：活动表、配置表、日志表（按需）

### 2.2 建表与 JPA 实体
- [x] 编写 SQL 建表脚本（或使用 JPA 自动 DDL，注意生产改为 validate）
- [x] 在项目中创建对应 JPA Entity，建立关联关系（OneToMany、ManyToOne 等）
  - 已实现：用户与权限（SysUser/SysRole/SysUserRole）、用户行为（UserFavorite/UserCartItem/GearRating/GearBrowseLog）、订单与推荐结果（OrderMain/OrderItem/RecommendResult）、装备（GearCategory/GearItem/GearTag/GearItemTag）、出行计划（TripPlan/TripPlanGear）及对应 Repository；社区实体待补。
- [x] 配置数据源、JPA、连接池（如 Hikari），本地可先用 H2 做单元测试

### 2.3 装备知识库与标签体系
- [x] 定义装备分类与标签（如：帐篷、睡袋、季节、预算区间、场景：徒步/露营/登山等）
- [x] 为后续「约束过滤」「基于内容推荐」提供可查询的维度

## 阶段五：推荐算法与接口（约 2–3 周，核心）

### 5.1 约束过滤（第一层）
- [x] 根据「出行计划」与用户输入生成约束：季节、预算区间、场景、人数等
- [x] 在装备表上做多维度过滤（Specification），得到候选集
- [x] 软约束：季节/场景为空时装备仍可入选

### 5.2 协同过滤（第二层）
- [x] 基于用户行为（收藏、评分、加购、购买）构建用户-装备关系
- [x] 实现基于用户的协同过滤（找相似用户，推荐他们喜欢的装备）
- [x] 从候选集中按协同过滤得分排序，可限制 Top-N

### 5.3 基于内容推荐（第三层）
- [x] 利用装备属性（标签、场景、季节）与计划/用户偏好
- [x] 计算装备与计划或用户偏好的相似度（Jaccard 标签重叠）
- [x] 与协同过滤结果加权融合，得到最终推荐列表

### 5.4 推荐服务与接口
- [x] 统一推荐入口：输入「计划 ID」或「用户 + 约束」，返回排序后的装备列表
- [x] 结果可缓存到 Redis（按 planId/userId 做 key，30 分钟过期）
- [ ] 管理员端：推荐策略配置、效果统计（若需论文数据）
