# 阶段一：需求与功能清单

依据开题报告《基于 Spring Boot 的户外运动装备推荐系统的设计与实现》整理，用于固化需求与模块边界。

---

## 1. 用户角色与权限

| 角色 | 说明 | 权限级别 |
|------|------|----------|
| **系统管理员** | 平台综合管理与技术维护 | 最高 |
| **注册用户** | 户外运动爱好者，使用推荐与装备相关功能 | 普通 |
| **装备管理员** | 负责装备信息维护，需申请与审核 | 管理端 |
| **专家** | 户外/装备领域专业人士，参与装备审核与专业内容 | 管理端 |

- 所有角色均需通过**注册 + 登录**成为系统用户；装备管理员与专家需**额外申请**并由管理员审核通过后获得对应角色。

---

## 2. 功能清单（必须实现 vs 可选）

### 2.1 系统管理员 — 必须实现

| 功能模块 | 具体功能 | 优先级 |
|----------|----------|--------|
| 用户与权限 | 查看/编辑/禁用用户，用户列表与搜索 | 必须 |
| 用户与权限 | 角色分配（普通用户 / 装备管理员 / 专家） | 必须 |
| 装备管理 | 装备信息增删改查、上下架、数据维护 | 必须 |
| 内容审核 | 评论、帖子、举报处理，敏感词/违规内容管理 | 必须 |
| 推荐系统 | 推荐策略配置、推荐效果查看（如点击/加购统计） | 必须 |
| 系统运营 | 基础配置、活动创建/编辑、简单数据统计 | 必须 |

### 2.2 注册用户 — 必须实现

| 功能模块 | 具体功能 | 优先级 |
|----------|----------|--------|
| 账号 | 注册、登录、个人信息查看/修改 | 必须 |
| 出行计划 | 创建/编辑/删除计划（目的地、时间、人数、预算、季节、活动类型等） | 必须 |
| 推荐 | 根据计划或约束获取**个性化装备推荐列表**，支持收藏、加购 | 必须 |
| 装备 | 装备列表、详情、多条件筛选、对比 | 必须 |
| 交易 | 加入购物车、下单流程（列表→下单→状态） | 必须 |
| 社区 | 发帖、评论、查看他人帖子与评论 | 必须 |
| 积分与等级 | 积分规则（发帖/评论/购买等）、等级展示、积分查询 | 建议实现 |

### 2.3 装备管理员 / 专家 — 必须实现

| 功能模块 | 具体功能 | 优先级 |
|----------|----------|--------|
| 身份 | 申请成为装备管理员/专家，等待管理员审核 | 必须 |
| 装备 | 装备信息维护（CRUD）、上下架（视权限配置） | 必须 |
| 内容 | 参与装备相关专业内容审核（若与管理员分工） | 可选 |

---

## 3. 各角色功能边界（简要）

- **系统管理员**：用户与角色、装备全生命周期、内容审核、推荐配置与运营、活动与基础数据统计。不参与日常装备编辑，以「管理」为主。
- **注册用户**：仅使用「计划 → 推荐 → 装备浏览/收藏/加购 → 下单 → 社区」全流程；不可管理其他用户或装备库。
- **装备管理员 / 专家**：在管理员授权下维护装备数据与（可选）部分审核；一般不分配用户管理、系统配置权限。

---

## 4. 核心业务模块（与后续阶段对应）

| 模块 | 阶段 | 说明 |
|------|------|------|
| 用户与认证 | 三 | 注册、登录、JWT、角色权限 |
| 装备管理 | 四 | 装备 CRUD、分类/标签、筛选、上下架 |
| 出行计划 | 四 | 计划 CRUD，作为推荐输入 |
| 推荐引擎 | 五 | 约束过滤 + 协同过滤 + 内容推荐，统一推荐接口 |
| 前端用户端 | 六 | 计划、推荐、装备、购物车、社区、个人中心 |
| 前端管理端 | 六 | 用户、装备、内容审核、推荐配置、运营 |
